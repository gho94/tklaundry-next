# 아키텍처 — 차세대 전환(TKLaundry Next)

> 목표: “매장에서 실제로 굴리는” 기준으로 단순하고 복구 가능한 구조를 만든다.

---

## 1. 시스템 구성(최소)

```
[ Flutter Windows(관리자) ]  --HTTP/JSON-->  [ Java API(Spring Boot) ]  --JDBC-->  [ MSSQL(기존) ]
```

### 1.1 배포/실행 옵션

- **옵션 A(권장, 단순)**: 앱과 API를 **같은 PC**에서 실행 (`localhost`)
- 옵션 B: 앱은 매장 PC, API는 같은 LAN의 PC에서 실행(관리/백업이 쉬운 경우)

> 병행 운영 초기에 옵션 A가 디버깅이 압도적으로 쉽다.

---

## 2. 저장소(모노레포) 구조

| 경로 | 역할 |
|------|------|
| `docs/` | 기획/설계/프로세스(이 문서 포함) |
| `backend/` | Java API |
| `app/` | Flutter Windows(관리자) |

---

## 3. 환경/설정 원칙(프로덕션 기준)

### 3.1 비밀값/로컬값 분리

- **절대 커밋 금지**: DB 비밀번호, 연결 문자열, 매장 환경 경로
- 원칙:
  - `application.properties` = 공용(안전한 값만)
  - `application-local.properties` = 로컬 개발(커밋 금지)
  - 필요 시 `application-store.properties` = 매장(커밋 금지, 별도 보관)

### 3.2 포트/주소

- API 기본 포트: `8080`(충돌 시 변경)
- 앱 Base URL:
  - 옵션 A: `http://127.0.0.1:8080`
  - 옵션 B: `http://<LAN_IP>:8080`

---

## 4. API 기본 규칙

- Prefix: `/api/...`
- JSON only
- 날짜/시간: ISO-8601(서버는 KST 기준 처리 방식 명시)
- 오류 응답 표준화(코드/메시지/traceId)

---

## 5. 데이터 접근(백엔드)

### 5.1 JDBC/SQL 원칙

- “기존 DB”에 대해 ORM(JPA)은 **초기엔 강제하지 않음**
  - 이유: 기존 스키마/규칙을 정확히 따라야 해서, 초기엔 SQL이 추적/대조가 쉬움
- P0 범위는 “조회/상태 변경 1개” 위주로 시작

### 5.2 트랜잭션 원칙

- 상태 변경(쓰기)에는 트랜잭션 적용
- 병행 운영 충돌 방지(필요 시):
  - 업데이트 조건에 “현재 상태” 포함(낙관적 잠금처럼 사용)
  - 또는 버전 컬럼/타임스탬프 기반 방어(필요 시)

---

## 6. 로깅/관측성

### 6.1 백엔드 로그

- 요청 단위 traceId 생성/전파
- 예외는 stacktrace 포함(로컬/운영 분리 가능)

### 6.2 앱 로그/오류 UX

- 사용자에게는 “조치 가능한 메시지”
- 로그에는 traceId/원인(HTTP status, endpoint, body 요약)

---

## 7. 향후 확장(범위 밖이지만 고려)

- Android/Web을 추가할 수는 있으나, 이 저장소의 성공 기준은 **Windows 관리자 병행 운영**이다.
- 플랫폼 추가는 Flutter에서 재생성 가능(코드 공유는 `lib/` 중심 유지).
