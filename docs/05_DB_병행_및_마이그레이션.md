# DB 병행 및 마이그레이션 — 차세대 전환(TKLaundry Next)

> 핵심: **기존 C#과 동일 MSSQL을 공유**하면서, “불일치/충돌”을 통제하고, 최종적으로 차세대로 전환한다.

---

## 1. 원칙(반드시 지킬 것)

- **동일 테이블에 동시 쓰기 금지**가 기본
- 병행 기간에는 “업무 단위”로 **쓰기 담당**을 명시한다
- 숫자 대조(건수/금액/상태)는 **매일** 가능해야 한다

---

## 2. 병행 운영 전략(선택지)

### 2.1 전략 A — 업무 단위 분리(권장)

- 예: “상태 변경(완료)”만 차세대가 쓰고, 나머지는 기존이 계속 씀
- 장점: 충돌 최소화, 위험 낮음
- 단점: 전환이 느릴 수 있음

### 2.2 전략 B — 기간/일자 단위 분리

- 예: 특정 날짜/기간의 신규 데이터만 차세대가 담당
- 장점: 점진 확대 쉬움
- 단점: 기존 화면/리포트가 기간 혼합 시 헷갈릴 수 있음

### 2.3 전략 C — 완전 전환(병행 최소)

- 일정 시점 이후 차세대만 사용
- 장점: 운영 단순
- 단점: 리스크 큼(되돌리기 어려움)

---

## 3. 쓰기 담당 표(필수)

> 아래 표를 “초안 → 확정”으로 바꾸고, 병행 기간에는 수정 시 이력을 남긴다.

| 업무/화면 단위 | 기존(C#) | 차세대 | 충돌 위험 | 비고 |
|---|---:|---:|---|---|
| 접수 목록 조회 | R | R | 낮음 | 둘 다 읽기만 |
| 접수 상세 조회 | R | R | 낮음 | 둘 다 읽기만 |
| 상태 변경(예: 완료) | - | W | 중 | P0 쓰기 1개로 시작 |
| 접수 등록 | W | - | 높음 | P1에서 전환 |
| 수납/결제 | W | - | 높음 | 규칙 복잡, 후순위 |

표기: `R`=read, `W`=write

---

## 4. 대조(정합성) 체크 — “매일” 할 것

### 4.1 대조 기준

- 기간: 당일/전일/지정 기간
- 비교 항목:
  - [ ] 총 건수
  - [ ] 상태별 건수
  - [ ] 총액 합계(필요 시 상태별)

### 4.2 대조 기록 양식(권장)

| 날짜 | 범위 | 기존 건수/총액 | 차세대 건수/총액 | 차이 | 원인/조치 |
|---|---|---|---|---|---|
| | | | | | |

---

## 5. DB 접속/비밀 관리

- 연결 문자열/계정/비밀번호는 **절대 커밋 금지**
- 로컬 개발용은 `application-local.properties`로 분리
- 매장 운영용 설정은 별도 보관(암호화/USB 등 현실적인 방식)

---

## 6. 스키마 변경 정책

병행 중 스키마 변경은 “양쪽에 영향”이므로 반드시 절차를 따른다.

- [ ] 변경 사유/영향 범위 문서화
- [ ] 백업 확보
- [ ] 적용 순서(기존 ↔ 차세대) 명시
- [ ] 대조 쿼리 업데이트

---

## 7. 전환/롤백(프로덕션 필수)

### 7.1 전환 조건

- [ ] P0 시나리오가 병행 운영에서 **N일** 연속 문제 없이 동작
- [ ] 대조 기준이 일치(허용오차 0 또는 사유 문서화)

### 7.2 전환 절차(초안)

- [ ] 전환일/전환 범위 고정
- [ ] 차세대 사용 범위 확대(쓰기 담당 표 업데이트)
- [ ] 문제 발생 시: 차세대만 중지하고 기존으로 운영(롤백)

### 7.3 롤백 절차(초안)

- [ ] API 중지
- [ ] 앱 사용 중지(바로가기 제거/안내)
- [ ] DB는 동일이므로 데이터 “되돌리기”가 필요하면 별도 스크립트(가능하면 설계로 예방)
