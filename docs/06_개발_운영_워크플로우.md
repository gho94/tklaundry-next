> 목표: 혼자 개발하더라도 “나중에 봐도 다시 굴릴 수 있는” 운영 수준을 만든다.

## 1. Git/브랜치 전략(단순)

- 기본 브랜치: `main`
- 작업 브랜치: `feat/*`, `fix/*`, `chore/*`
- 커밋 메시지(권장):
  - `feat: ...`
  - `fix: ...`
  - `docs: ...`

---

## 2. 로컬 실행(개발)

### 2.1 백엔드

```powershell
cd backend\tklaundry
.\gradlew.bat bootRun
```

헬스:
- `GET http://localhost:8080/api/health` → `ok`

### 2.2 앱(Flutter Windows)

```powershell
cd app\tklaundry
flutter run -d windows
```

---

## 3. 환경 분리(필수)

### 3.1 백엔드 설정 파일 정책

- `application.properties` : 공용(안전한 값만)
- `application-local.properties` : 로컬(커밋 금지)
- `application-store.properties` : 매장(커밋 금지)

### 3.2 커밋 금지 대상(예)

- DB 계정/비밀번호/서버 주소
- `application-*.properties` 중 운영/로컬 비밀 포함 파일
- Flutter `build/`, `.dart_tool/`

---

## 4. 릴리즈/배포(초안)

### 4.1 백엔드 배포 방식(선택)

- 옵션 A: 매장 PC에서 `bootRun`으로 시작(초기/실험)
- 옵션 B(권장): `jar` 빌드 후 서비스처럼 실행

```powershell
.\gradlew.bat bootJar
```

산출물: `build\libs\*.jar`

### 4.2 앱 배포

- 초기엔 개발 PC/매장 PC에 직접 배포
- (추후) 설치 파일/업데이트 방식은 별도 문서화
