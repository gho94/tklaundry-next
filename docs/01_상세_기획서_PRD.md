# PRD — 세탁소 관리 시스템 (차세대 전환)

> **프로젝트명**: TKLaundry Next  
> **작성/수정**: Geonho / 2026-04-27 / 초안  
> **핵심**: 기존 `TKLaundry`(C#)와 **동일한 업무 프로세스**를 **Java API + Flutter Windows**로 재구현하고, **동일 MSSQL** 기반으로 **병행 운영** 후 점진 전환한다.

---

## 1. 배경/문제

- 기존 앱은 **DevExpress 종속** 및 UI/유지보수 제약이 있음
- 향후 유지보수·협업을 위해 **오픈소스 친화 스택**으로 전환 필요
- 전환 리스크를 줄이려면 “한 번에 갈아엎기”가 아니라 **병행 운영으로 검증**해야 함

---

## 2. 목표/비목표

### 2.1 목표

- **동일 업무 프로세스**를 차세대에서 처리 가능
- **병행 운영 기간** 동안 핵심 지표(건수/금액/상태)가 **기존과 일치**함을 증명
- 차세대가 “혼자서” 돌아갈 수 있도록 **프로덕션 운영 기준(백업/롤백/로그/권한)**을 갖춤

### 2.2 비목표(이 PRD 범위 밖)

- 고객용 앱/웹/모바일(관리자 외 사용자)
- 클라우드 마이그레이션/멀티테넌시/공인 API 공개
- 결제/외부 연동(필요 시 별도 PRD)

---

## 3. 사용자/권한

| 역할 | 설명 | 권한 |
|------|------|------|
| 관리자(운영) | 매장 운영자(본인) | 전체(초기 1-role) |

> 초기에는 **권한 1종(관리자)**으로 단순화하고, 필요 시 “직원/조회 전용”을 추가한다.

---

## 4. 운영 시나리오(병행)

### 4.1 설치/실행 형태(현실 기준)

- **매장 PC**: Flutter Windows 앱 실행
- **API**: 같은 PC(로컬) 또는 같은 LAN의 별도 PC에서 실행(선택)
- **DB**: 기존 MSSQL **동일 인스턴스/동일 DB** 사용

### 4.2 병행 운영 원칙

- “쓰기 충돌”을 피하기 위해 **업무 단위로 쓰기 담당을 명시**한다(문서: `05_DB_병행_및_마이그레이션.md`)
- 병행 기간 중에는 “대조 쿼리/대조 화면”으로 **일치 여부를 매일 확인**한다

---

## 5. 범위(기능) — P0/P1/P2

> 아래는 “기존 앱과 동일한 화면/업무 흐름” 기준으로 작성한다.  
> 실제 테이블/컬럼 매핑은 `05_DB_병행_및_마이그레이션.md`에서 확정.

### 5.1 P0 (병행 운영 가능 최소)

- [ ] **주요 조회 1개**: 오늘/기간별 접수 목록(핵심 필터 포함)
- [ ] **상세 조회**: 접수 1건 상세(상태/금액/메모)
- [ ] **업무 상태 변경 1개**: 예) “완료 처리” 또는 “출고 처리”(업무 1개만)
- [ ] **핵심 합계 대조**: 건수/총액(기존과 비교 가능한 기준)
- [ ] **오류/로그**: 앱/서버 오류를 “원인 추적 가능” 수준으로 남김

### 5.2 P1 (P0 안정화 후 확장)

- [ ] 접수 등록(신규)
- [ ] 결제/수납 처리(기존 흐름 유지)
- [ ] 검색/필터 고도화(고객/전화/태그 등)

### 5.3 P2 (선택/개선)

- [ ] UX 개선(단축키/대량 처리)
- [ ] 인쇄/라벨(필요 시)
- [ ] 권한 분리(직원/조회 전용)

---

## 6. 데이터/정합성 요구사항(가장 중요)

### 6.1 정합성 정의(“일치”의 기준)

- 동일 기간/동일 조건에서:
  - [ ] **건수**가 일치
  - [ ] **총액/합계**가 일치
  - [ ] **상태별 분포**(예: 접수/진행/완료)가 일치

> 수동 검증: `10_유저플로우_테스트.md` (절차별 체크)

### 6.2 충돌 방지(쓰기 담당)

- 병행 기간에는 “같은 테이블/같은 컬럼”을 양쪽이 동시에 바꾸지 않도록:
  - [ ] 업무 단위로 **쓰기 담당(기존/차세대)**를 표로 고정
  - [ ] 불가피하면 DB 제약/트리거/버전 컬럼 등으로 방어(필요 시)

---

## 7. 비기능 요구사항(프로덕션 수준)

### 7.1 안정성/복구

- [ ] 장애 시 **차세대만 중지**하고 기존으로 계속 운영 가능(롤백)
- [ ] DB 백업/복구 절차 문서화

### 7.2 보안/비밀 관리

- [ ] DB 계정/비밀번호는 **절대 git 커밋 금지**
- [ ] 로컬 설정은 `application-local.properties` 등 프로필 분리

### 7.3 관측성(로그)

- [ ] API 요청/응답(요약) + 예외 스택트레이스 로깅
- [ ] 앱에서 서버 오류를 식별 가능한 메시지로 표시(사용자에게는 친절, 로그에는 상세)

### 7.4 성능

- [ ] “하루 운영” 수준에서 목록/검색이 체감 지연 없이 동작
- [ ] 느린 쿼리(인덱스) 이슈는 병행 중에 우선 해결

---

## 8. 기술 스택/원칙

| 계층 | 기술 | 원칙 |
|---|---|---|
| App | Flutter Windows | “관리자용 데스크톱”에 최적화 |
| API | Java + Spring Boot | REST/JSON, 내부용 |
| DB | MSSQL(기존) | 동일 DB 공유(병행) |

---

## 9. 성공 기준(수락 기준)

- [ ] P0 기능으로 실제 업무를 1일 이상 병행 운영
- [ ] 대조 기준(건수/금액/상태)이 일치(허용 오차 0 또는 사유 문서화)
- [ ] 장애 시 복구/롤백 절차가 문서로 존재하고 1회 이상 리허설
